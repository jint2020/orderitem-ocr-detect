# 移动端订单页 OCR 需求评估与落地计划

## 1. 背景与目标

用户在前端上传移动端 App 订单页面截图，或手机拍摄的移动端 App 订单页面照片。页面主体信息以 cell 表单形式展示，通常为 `label: value` 或左右两列字段结构。系统需要识别并抽取以下字段，返回给前端供用户确认：

- `号码`
- `日期`
- `姓名`
- `套餐信息`

服务器性能有限，第一阶段不推荐使用多模态大语言模型直接做 OCR 或字段抽取。优先采用轻量、可解释、可部署、可持续优化的传统 OCR 流程。

## 2. 推荐结论

第一版推荐采用：

```text
图片预处理 -> OCR文字检测/识别 -> cell行结构重建 -> label-value字段抽取 -> 置信度校验 -> 前端确认 -> 用户修正回流
```

模型优先选择 `PaddleOCR PP-OCRv6/PP-OCRv5` 中文检测与中文识别模型；如果部署侧希望进一步减轻运行时依赖，可评估 `RapidOCR + ONNXRuntime`。

第一阶段不建议先训练模型。应先收集真实样本，跑通 OCR + 规则抽取闭环，量化错误类型后再决定是否微调检测模型、识别模型或引入轻量 KIE 模型。

## 3. 流程决定

### 3.1 输入范围

第一阶段建议限定输入范围：

- 移动端 App 订单页面截图。
- 手机拍摄的移动端 App 订单页照片。
- 页面信息以 cell 表单展示，字段名和值在同一行或相邻区域。
- 目标字段固定为 `号码`、`日期`、`姓名`、`套餐信息`。

暂不覆盖：

- 完全开放的任意业务页面。
- 多页 PDF 或长截图跨屏拼接。
- 表格、票据、证件等非订单页场景。
- 使用多模态大语言模型兜底识别。

### 3.2 后端处理流程

1. **上传与校验**
   - 限制图片格式：`jpg`、`jpeg`、`png`、`webp`。
   - 限制图片大小和分辨率，避免超大图片压垮服务。
   - 生成请求 ID，便于日志追踪和错误样本回收。

2. **图像预处理**
   - 读取 EXIF 方向并自动旋转。
   - 对拍照图片做轻量纠偏。
   - 按模型推荐尺寸缩放，保留长边细节。
   - 对低清晰度图片做基础锐化或对比度增强。
   - 对明显非订单页面或极度模糊图片直接返回低置信度结果。

3. **OCR 检测与识别**
   - 文本检测模型输出文字框。
   - 文本识别模型输出文字内容和置信度。
   - 保留每段文字的 `text`、`box`、`score`。

4. **版面结构重建**
   - 按 y 坐标将文字框聚类成行。
   - 同一行内按 x 坐标排序。
   - 合并距离很近的碎片文本。
   - 识别 `label` 和 `value` 的空间关系。

5. **字段抽取**
   - 建立字段别名字典，例如：
     - `号码`：`号码`、`手机号`、`联系电话`、`业务号码`、`入网号码`
     - `日期`：`日期`、`办理时间`、`订单时间`、`生效时间`
     - `姓名`：`姓名`、`客户姓名`、`用户姓名`
     - `套餐信息`：`套餐`、`套餐信息`、`资费套餐`、`产品名称`
   - 优先读取同一行右侧 value。
   - 同行没有 value 时，读取下一行或空间距离最近的文本。
   - 对每个字段做格式校验和置信度融合。

6. **结果返回**
   - 返回字段值、字段置信度、OCR 原始框、候选值。
   - 前端展示识别结果，让用户确认或修正。
   - 低置信度字段需要明显提示用户核对。

7. **数据回流**
   - 记录 OCR 原始输出、系统抽取值、用户确认值、用户修正值。
   - 将失败样本按错误类型归档，用于规则优化和后续训练。

## 4. 模型选择

### 4.1 推荐方案：PaddleOCR

优先评估 `PaddleOCR PP-OCRv6/PP-OCRv5` 中文检测模型和中文识别模型。

优点：

- 中文 OCR 场景成熟。
- 支持文本检测、文本识别、方向分类等常用能力。
- 可在 CPU 上部署，适合服务器性能受限场景。
- 具备后续微调训练链路。
- 社区资料和部署案例较多。

适用场景：

- 移动端截图。
- 拍照后的中文表单文本。
- 固定字段抽取。
- 对定位框有要求，需要前端展示原图位置。

### 4.2 轻量部署备选：RapidOCR + ONNXRuntime

当服务端希望降低 Paddle 运行时依赖，或希望使用 ONNXRuntime 做轻量推理时，可评估 RapidOCR。

优点：

- 部署相对轻。
- ONNXRuntime 推理链路简单。
- 适合 CPU 资源有限的服务。

风险：

- 模型版本、训练链路、问题排查能力通常不如直接使用 PaddleOCR 完整。
- 如果后续需要深度训练和定制，仍可能回到 PaddleOCR 工具链。

### 4.3 暂不推荐：多模态大语言模型 OCR

第一阶段不推荐使用多模态大语言模型直接识别订单字段。

原因：

- 推理成本和延迟较高。
- 服务端资源压力更大。
- 文本定位框输出不稳定，不利于前端高亮确认。
- 对固定字段、结构化页面来说性价比不高。
- 可能出现幻觉或格式不稳定，需要额外校验。

只有当后续页面类型高度开放、传统 OCR + 规则/KIE 无法覆盖，且业务能接受成本和延迟时，才考虑作为离线质检或人工辅助工具，而不是主链路。

## 5. 模型训练计划

### 5.1 第一阶段：不训练，先评估

先使用现成 OCR 模型建立 POC，不要直接进入训练。

目标：

- 验证 OCR 对真实订单页的基础识别能力。
- 判断错误主要来自图片质量、文字检测、文字识别、版面配对还是字段规则。
- 建立字段级准确率基线。

样本建议：

- POC：100-300 张真实图片。
- MVP：500-1000 张真实图片。
- 覆盖截图、拍照、倾斜、反光、模糊、深色模式、不同字号、不同手机分辨率。

### 5.2 标注策略

第一阶段只标注最终字段值：

```json
{
  "image_id": "sample_0001",
  "号码": "13800001234",
  "日期": "2026-06-14",
  "姓名": "张三",
  "套餐信息": "5G畅享套餐129元"
}
```

当需要训练检测或识别模型时，再补充文字框级标注。

当需要训练 KIE 模型时，再补充 label-value 关系标注。

这样可以避免一开始就投入过重的标注成本。

### 5.3 训练触发条件

按错误类型决定训练方向：

| 错误类型 | 表现 | 优先处理方式 |
| --- | --- | --- |
| 图片质量差 | 模糊、反光、倾斜导致字段不可读 | 前端上传提示 + 图像预处理 |
| 检测漏框 | 页面上有字但 OCR 没框出来 | 微调文本检测模型 |
| 识别错误 | 框正确但文字识别错 | 微调文本识别模型 |
| 字段配错 | OCR 文字正确，但 value 匹配到错误 label | 优化版面规则，必要时训练 KIE |
| 字段别名缺失 | label 未被字段字典命中 | 扩充字段别名字典 |
| 套餐信息截断 | 长文本分多行或被拆碎 | 优化行合并和邻近文本合并 |

### 5.4 训练优先级

训练优先级建议如下：

1. **先优化规则和预处理**
   - 成本最低，见效最快。
   - 适合固定或少量模板。

2. **再微调识别模型**
   - 当号码、姓名、套餐文字经常被识别错时进行。
   - 需要文字行图片和对应文本标注。

3. **再微调检测模型**
   - 当拍照图片漏检、斜拍漏检明显时进行。
   - 需要文字框标注。

4. **最后引入轻量 KIE**
   - 当页面模板增多、规则维护成本升高时进行。
   - 目标是学习 label-value 关系，而不是替代 OCR。

## 6. 评估指标

### 6.1 字段级指标

核心指标应按字段统计：

- `号码`准确率
- `日期`准确率
- `姓名`准确率
- `套餐信息`准确率
- 四字段全部正确率
- 低置信度召回率
- 用户修正率

建议 POC 目标：

| 输入类型 | 单字段准确率目标 | 四字段全部正确率目标 |
| --- | --- | --- |
| 清晰截图 | >= 95% | >= 90% |
| 普通拍照 | >= 85% | >= 75% |
| 倾斜/反光/模糊照片 | 不强制自动通过 | 应进入人工确认 |

### 6.2 置信度策略

建议字段置信度由以下因素融合：

- OCR 文本识别置信度。
- label 命中置信度。
- label 与 value 的空间距离。
- value 格式校验结果。
- 是否存在多个候选值冲突。

建议策略：

- `confidence >= 0.90`：默认可信，但仍展示给用户确认。
- `0.70 <= confidence < 0.90`：展示为需重点核对。
- `confidence < 0.70`：不自动填入或标记为识别失败。

## 7. 里程碑计划

### 阶段一：POC 验证

周期建议：1-2 周。

交付物：

- 基础 OCR 服务。
- 图片预处理流程。
- label-value 抽取规则。
- 四个字段的 JSON 返回。
- 100-300 张样本的评估报告。

验收标准：

- 清晰截图可稳定识别。
- 字段结果可返回前端确认。
- 能输出错误类型统计。

### 阶段二：MVP 闭环

周期建议：2-4 周。

交付物：

- 前端确认和修正流程。
- 用户修正数据回流。
- 字段置信度策略。
- 错误样本归档机制。
- 500-1000 张样本评估报告。

验收标准：

- 支持截图和普通拍照图片。
- 低置信度字段能被前端提示。
- 用户修正数据可用于后续优化。

### 阶段三：定向优化与训练

周期建议：按错误类型决定。

交付物：

- 规则优化版本。
- 必要时微调识别模型或检测模型。
- 页面模板增多后可评估轻量 KIE。

验收标准：

- 明确证明训练收益高于规则优化。
- 训练后字段级准确率有可量化提升。
- 推理延迟和资源消耗仍满足服务器限制。

## 8. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 拍照图片质量差 | OCR 识别率下降 | 前端拍摄提示、模糊检测、低置信度确认 |
| 页面模板变化 | 字段配对错误 | 字段别名字典、模板版本统计、错误回流 |
| 套餐信息过长 | 文本截断或合并错误 | 多行合并、候选值保留、前端确认 |
| 姓名脱敏 | 无法判断真实姓名 | 明确业务是否接受脱敏值 |
| 号码类型不统一 | 手机号、宽带号、业务号规则不同 | 建立号码类型识别和校验规则 |
| OCR 模型延迟 | 服务吞吐下降 | 使用轻量模型、ONNXRuntime、异步队列、限流 |
| 过早训练 | 标注成本高且收益不确定 | 先做评估集和错误归因 |

## 9. 待确认问题

后续进入实施前，需要确认：

1. 订单页来源是固定一个 App，还是多个 App。
2. 页面模板数量大约有多少。
3. `号码`具体指手机号、宽带号、订单号，还是多种号码。
4. `日期`指订单日期、办理日期、生效日期，还是页面中某个固定 label。
5. 姓名是否可能脱敏，例如 `张*`。
6. 套餐信息是否需要完整原文，还是只需要套餐名称。
7. 是否允许保存用户上传图片和修正结果用于训练。
8. 服务端 CPU、内存、并发、延迟目标是多少。

## 10. 最终建议

第一阶段目标应是跑通可解释闭环，而不是追求端到端智能抽取。

推荐路线：

```text
PaddleOCR / RapidOCR
  + 图像预处理
  + cell表单版面解析
  + 字段别名和格式规则
  + 前端人工确认
  + 修正数据回流
```

当真实样本证明规则无法覆盖时，再按错误类型定向训练 OCR 检测、OCR 识别或轻量 KIE 模型。这样可以在服务器资源有限的情况下，用较低成本获得稳定、可解释、可持续优化的 OCR 字段抽取能力。
